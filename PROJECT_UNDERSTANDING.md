## 1. High-level architecture

- **Framework**: TanStack Start (file-based router, React 19, Vite 8, SSR via Nitro). Deployed to Vercel (`nitro: { preset: "vercel" }` in `vite.config.ts`). Scaffolded and still synced through **Lovable** (`AGENTS.md`, `.lovable/project.json` ŌĆö pushes to the connected branch sync back into the Lovable editor).
- **State**: no traditional backend/API layer. All business data lives in **Zustand stores with `persist` middleware** (`localStorage`). There is no REST/GraphQL API ŌĆö UI reads/writes stores directly; the only server round-trips are TanStack `createServerFn` calls used for AI, Places, Tally, and sync.
- **Sync**: a custom layer (`src/lib/sync.ts`) intercepts `localStorage.setItem` for a fixed set of store keys and debounce-pushes each store's full JSON blob to **Turso (libSQL)** via two server functions (`pushState`/`pullState` in `src/lib/db.functions.ts`). One shared workspace (`"default""`), gated by a single passphrase (`ATLAS_PASSPHRASE` env var) ŌĆö effectively single-tenant, no per-user accounts, no RBAC.
- **AI**: all AI calls are server functions in `src/lib/ai.functions.ts` (1489 lines) that call Groq / OpenRouter / Google Places directly via `fetch` ŌĆö no AI SDK, no LangChain, no streaming.
- **Auth**: `AuthGate.tsx` ŌĆö a single shared-passphrase lock screen, or "use offline" (local-only, no sync). Not user auth in any conventional sense.

## 3. "Database schema"

There is no relational schema. Turso holds one table, created lazily:

```sql
CREATE TABLE IF NOT EXISTS store (
  workspace TEXT NOT NULL,
  key TEXT NOT NULL,
  value TEXT NOT NULL,   -- full JSON blob of one Zustand store
  updated_at INTEGER NOT NULL,
  PRIMARY KEY (workspace, key)
)
```

One row per synced store (`atlas-crm`, `atlas-tasks`, `atlas-ledger`, `atlas-inventory`, `atlas-barter`, `atlas-accounts`, `atlas-brand-deals`, `atlas-settings`, `atlas-credits`, `atlas-warehouse`, `atlas-bookings`). `atlas-google` (OAuth token) and the passphrase are intentionally device-local, not synced.

The de-facto "schema" is the TypeScript interfaces in `stores.ts`/`crm.ts`/`ledger.ts`: `Site`, `Booking`, `WarehouseItem`+`StockMovement`, `Partner`, `BarterDeal`, `Invoice`+`InvoiceLine`+`Payment`, `Settings`, `UsageEntry`, `BrandDeal`+`MediaPlanItem`, `Customer`, `Task`, `JournalEntry`. No migrations exist or are needed ŌĆö shape changes just change what's in the JSON blob.
